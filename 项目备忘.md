## 2026-05-25 机内轻量音质优化说明

这轮写进固件的是轻量实时处理，不是神经网络降噪，也不是网上大模型算法。目标是让 Cardputer 这颗小麦克风录出来的人声更稳、更少刺耳高频和底噪，同时不能让 ESP32-S3 的录音链路卡顿。

依据：
- Cardputer-Adv 没有 PSRAM，不能上 Espressif ESP-SR 这类较重的专业降噪链路。
- 之前样本显示，按键/手指摩擦声通常 RMS 不一定高，但相邻采样变化量大、高频比例高；人声 RMS 更稳定，频谱重心相对低。
- 用户主要需求是清晰语音、可在机器本机播放、并能提升语音转文字；所以原始 WAV 保留，机内只做保守轻处理，服务器另做 ASR-clean。

固件当前 `processMicBuffer()` 思路：
- 先估计本帧 RMS 和相邻采样平均差分 `avgDiff`。
- 用 `rms < 520 && avgDiff > 120 && avgDiff/rms 较高` 判定疑似摩擦/细碎高频噪声。
- 维护一个很慢更新的 `noiseRms`，只在安静且不像摩擦的帧里学习底噪。
- 对接近底噪的帧做轻微 soft gate，不硬切静音，避免人声一卡一卡。
- 每个样本做 DC/high-pass 去偏移，再走一阶低通 `lpfCoef`。
- 疑似摩擦时减少高频混合 `hiMixQ8`，正常人声尽量保留高频清晰度。
- 最后用 `tanhf` 做软限幅，防止大声或碰撞时爆裂。

当前取舍：
- 优点：实时、轻量、不需要新库，录音不容易卡；比强门限/强 AGC 更自然。
- 缺点：不是专业降噪，环境噪声很大时只能改善，不能“抹干净”；参数过强会让人声发闷或断续。
- 下一步如果继续优化，优先收集三类样本：安静人声、有摩擦的人声、环境音。不要只凭单条录音盲调。

## 2026-05-25 服务器 ASR-clean 自适应门限版

服务器端 `buildAsrCleanWav()` 已从固定噪声门升级为“自适应底噪 + 平滑开关门”：
- 先扫描整条录音的帧 RMS，取低 20% 分位估计 `noiseFloorRms`。
- 实际识别门限 `adaptiveGateRms = max(手动 noiseGateRms, noiseFloorRms * 2.25 + 80)`，上限 1800，避免环境底噪变高时固定门限失效。
- 帧级 gate gain 先平滑：开门快、关门慢，再做一个轻微反向前视，保护字头。
- 样本级 gate gain 继续平滑，避免门限跳变造成“咔哒”或字尾被切。
- metadata 新增 `noiseFloorRms`、`adaptiveGateRms`、`minGateGain`，方便后台看这条录音到底被门限处理了多少。

部署备份：服务器 `/opt/cardputer-voice/server.js.bak-20260525-adaptive-asr-clean`。这版只影响服务器生成的 `clean-for-asr.wav` 和后续 DashScope 转写，不影响机器端原始录音。

## 2026-05-25 REC_0007 实测结论

测试录音：`20260525_225500_cardputer-001_REC_0007.wav`，时长约 45.38 秒。原始音频 DashScope 识别整体可用，错误主要集中在最后一句；ASR-clean 版反而把“请记录”识别成“请借入”，最后一句也更偏，所以这条样本最终已切回 `asrSource=raw`。

本次 ASR-clean 指标：`noiseFloorRms=5392`、`adaptiveGateRms=1800`、`gatedFrames=0/2269`。含义是这条录音整体环境/人声能量都很高，自适应门限没有真正开门降噪，后面的高通、预加重、压缩和归一化可能改变了语音特征，导致识别不如原始版。结论：服务器默认仍应保留 raw 优先；ASR-clean 只作为对比实验，不要自动替代最终转写，除非后续多条样本证明它稳定更准。

后续优化方向：先收集三类样本（安静人声、普通环境人声、真实环境音），每条都同时比较 raw 与 ASR-clean 的转写错误；如果要继续优化 ASR-clean，优先做“轻预处理/不压缩”的版本，而不是继续加大降噪强度。

## 2026-05-25 说话人分离功能

已接入 DashScope/Paraformer 的 `speaker_count` 参数。后台“转写设置”页新增“说话人数量”：默认“不分角色”，录对话/采访/会议时可选 2、3、4 人后重跑完整转写。服务器会把 DashScope 返回的 `speaker_id` 解析成：


## 2026-05-25 说话人口令自动触发

已实现“开头或结尾说口令，服务器自动二次重跑分角色转写”。流程：第一次上传仍按默认 raw 转写；服务器扫描转写文本开头 500 字和结尾 500 字，如果检测到人数口令，且当前还没分角色，就自动设置 `asrSpeakerCountRequested` 并重跑一次 DashScope，最终 memo 使用分角色文本。用 `autoSpeakerRetryDone` 防止重复循环。

后台会显示“已按录音口令自动启用 N 人分角色”。注意：口令需要被第一次 ASR 识别出来才会触发；如果口令在噪声里没识别准，仍可去后台手动选择人数重跑。

## 2026-05-26 说话人自动命名

已实现稳态自动命名：在分角色文本中，如果某个说话人自己的行明确出现“我是X / 我叫X / 我的名字叫X / 这里是X”，服务端会建立安全映射，把后续 `说话人1：` 替换为 `X：`。为了避免误标，别名必须较短、只含中文/英文/数字/点/下划线/短横线，且不能是“我们/你们/测试/对话”等普通词。

**2026-05-24 录音音质/按键摩擦处理结论**:
- 当前可用版本保留 GitHub 基线实时录音链路: `REC_RATE=16000`, `REC_N=256`, `recGain=50`, `mc.magnification=1`, `mc.noise_filter_level=3`, `processMicBuffer()` 仍是软件增益 + `lpf` 系数 `160` + `tanhf` 软限幅。
- 新增保存后轻量按键摩擦处理 `suppressKeyFriction(path, effData)`: 录完写好 WAV 后扫描一遍 PCM, 只处理"低 RMS + 高相邻采样变化量"的帧。
- 样本来源: `REC_0399.wav`=纯按键/手指摩擦, `REC_0400.wav`=说话夹杂摩擦。分析结论: 摩擦声 RMS 低(A线不明显), 但高频比例/相邻采样变化量高; 人声帧 RMS 高、频谱重心更低。
- 当前规则: `rms < 1900 && avgDiff > 140 && avgDiff*256 > rms*90` 或 `rms < 2200 && avgDiff > 300` 判为摩擦; 命中后 `hold=2` 个缓冲。
- 当前处理方式: 不压整块音量, 只削高频刮擦分量。每帧做一阶低通 `lp += (sample-lp)*96/256`, 分离 `hi=sample-lp`, 输出 `lp + hi*64/256`。这样比旧方案自然, 不会把说话变成断断续续。
- 明确踩坑: 不要再用强实时 AGC/强语音门/长时间整块衰减处理摩擦或碰撞; 这些会让人声变糊、断续, 且按键摩擦不一定会被 A线/RMS 捕捉到。

**2026-05-25 录音处理新方向：人耳播放版和机器识别版分开**:
- 以后不要把“小机器播放出来好听”和“阿里云语音识别更准”混成一个目标。它们是两套处理。
- 小机器端目标：录得稳、播放舒服、少刺耳、少刮擦、不要爆音。算法要轻，不能占太多资源，也不要让录音链路变脆。
- 服务器端目标：给语音转文字用。可以不好听，但要让人声更清楚、字头字尾别被切掉、识别更准。
- 原始 WAV 永远保留。推荐流程：`raw.wav` 原始录音 → 小机器/电脑预览用“人耳版处理” → 服务器生成 `clean_for_asr.wav` 给 DashScope → 保存转写和 flomo memo。
- 小机器算法可以拆成单独文件，比如 `audio_process.h/.cpp`，把实时轻处理、按键摩擦处理、参数结构从 `toolbox.ino` 里分出来。
- 电脑网页优先调“参数”，不要每次推 C++ 源码。调好后通过 Wi-Fi 推一份配置到小机器 SD 卡，例如 `/UPLOAD/audio.cfg` 或 `/CONFIG/audio.json`。大改算法时再重新编译烧录。
- 网页端要能 A/B 对比：原始录音、小机器播放版、服务器识别版。用户能听、能看波形、能看到哪些地方被判为刮擦声。
- 服务器处理不能是黑箱。网页后台要显示每条录音的处理步骤、使用的参数版本、处理后的文件、阿里云转写结果、最终发到 flomo 的文本。
- 后续可做 AI 自动调参闭环：录音 → 上传 → 网页/服务器分析 → 调参数 → Wi-Fi 推到小机器 → 再录音验证。第一阶段先人工网页调参，稳定后再自动化。

## 一、这是什么
给 M5Stack **Cardputer-Adv** 写的一个"工具箱"固件(Arduino/C++)。
- 风格: 纯黑底 + 黑客帝国荧光绿 + 中文界面。
- 主页: `[空格] 录音`  `[回车] 播放最近`。
- 录音机: 实时绿色声波(示波器), 录音中 **W 调大 / S 调小** 音量, **其他键停**。
- 录完结果页: `[回车] 播放`  `[N] 降噪`(FFT 谱减法)  其他键返回。
- 录音存成 16kHz 单声道 WAV, 放在 SD 卡 `/REC/REC_xxxx.wav`。

## 三、怎么编译 + 烧录(核心流程)
1. 用**能传数据的 USB-C 线**把机器连到电脑(劣质/充电线会"无法识别 USB", 换线!)。
2. 电源开关 ON。
3. 在 PowerShell 里跑(端口一般是 COM3, 不确定先用 `arduino-cli board list` 查):
```
$cli="C:\cardputer\tools\arduino-cli\arduino-cli.exe"
& $cli compile --fqbn esp32:esp32:m5stack_cardputer:FlashSize=8M,PartitionScheme=default_8MB C:\cardputer\sketches\toolbox
& $cli upload  --fqbn esp32:esp32:m5stack_cardputer:FlashSize=8M,PartitionScheme=default_8MB -p COM3 C:\cardputer\sketches\toolbox
```
- **开发板型号(FQBN)**: `esp32:esp32:m5stack_cardputer:FlashSize=8M,PartitionScheme=default_8MB`(8MB Flash + 3MB APP 分区, 给 Wi-Fi 上传留空间)。
- 盒子层联网配置在 SD 卡 `/UPLOAD/net.txt`；除 `ssid/password/url/token/device` 外，时钟可选 `ntp=pool.ntp.org`、`tz=8`。联网成功后盒子层会做 NTP 校时，上传时可带 `X-Recorded-At`。
- 长配置不在小机器上手敲；电脑打开 `toolbox/NET_CONFIG_HELPER.html` 填服务器 URL/token，然后生成/下载 `net.txt` 放到 SD 卡 `/UPLOAD/`。Wi-Fi 名/密码可在小机器端 `Wi-Fi` 应用里现场修改。
- 如果烧不进/认不到: 进**下载模式** = 电源 OFF → 按住 G0 → 通电 → 松开 G0, 再烧。
- 看程序崩溃日志: 用串口工具连 COM3 @115200, 会打印 `Guru Meditation` 等。

## 四、关键的坑和决定(很重要, 别重复踩)
1. **核心必须用 esp32 3.1.3, 不能用 3.3.x!**
   - 3.3.8(最新)上**麦克风录不了音**(录到静音/平线), 这是新核心的 I2S 录音回归 bug。
   - 3.1.3(IDF 5.3)麦克风正常。降级方法见 `C:\cardputer\prefetch313.ps1`。
2. **键盘库被改成"轮询"了**(去掉了 GPIO 中断)。
   - 文件: `...\M5Cardputer\...\KeyboardReader\TCA8418.cpp`。
   - 原因: 原版用 GPIO 中断, 在 3.1.3 上开机会撞爆系统 ipc1 任务栈 → 不停重启/闪屏。
   - ⚠️ **如果重装 M5Cardputer 库, 这个改动会丢, 要重新改**(把 begin() 里的 attachInterrupt 删掉、update() 改成 `while(available()) getEvent...` 轮询)。
3. **开机会"预热"麦克风**(空跑一次录音会话 + 暖一下 SD), 否则**冷启动后第一次录音音量小、断续**。
4. **国内下载慢**: esp32 核心用 Jihulab 镜像 + 从 `dl.espressif.cn` 用 curl 高速预取到缓存, 再让 arduino-cli 从缓存装(arduino-cli 自带下载器对国内 CDN 很慢)。
5. **这颗芯片(ESP32-S3FN8)没有 PSRAM** → 跑不了 Espressif ESP-SR 专业降噪。所以只能用自写的轻量谱减法降噪。
6. 喇叭是 1W 小喇叭, 本身就小声; **要大声清晰用 3.5mm 耳机**。

## 五、可调参数(都在 toolbox.ino 里)
| 参数 | 位置 | 作用 |
|---|---|---|
| `recGain`(默认 50) | 文件顶部 | 录音音量(也可录音时 W/S 现场调) |
| 低通系数 `160` | recorderApp 的 lpf 那行 | 越小越闷、压高频底噪越多 |
| `HEAD_SKIP`(13) | recorderApp | 掐头时长(去开始键声) |
| `tailTrim`(REC_RATE/4) | 收尾处 | 掐尾时长(去结束键声) |
| 软限幅 `tanhf` | 每采样处理 | 防大声"哒哒"爆裂 |
| `suppressKeyFriction()` | 保存后处理 | 按键/手指摩擦抑制: 低 RMS + 高 avgDiff 帧只削高频, 不整块降音量 |
| 降噪 `ALPHA`(3.0)/`BETA`(0.05) | noiseReduce() | 谱减强度/谱底; ALPHA 越大降噪越狠但易有"水声" |
| 喇叭 DAC `0x32` | playWav() | 回放音量微调 |

## 2026-05-26 固件端机器底噪前处理 v1

背景：服务器端 v16 频谱机器噪声指纹已经把网页试听里的底噪压得很干净，但它依赖 FFT/STFT 和参考噪声谱，不适合直接搬进 Cardputer 的实时录音内圈。机器本体没有 PSRAM，录音循环还要同时服务 SD 写入和 UI 刷新，所以固件端先做轻量源头治理，而不是把服务器算法硬塞进去。

本次固件改动只影响 `toolbox/toolbox.ino` 的实时录音链路，不覆盖已有 WAV，不烧录机器。改动点：
- 默认 `recGain` 从 `50` 降到 `36`，减少把机器本底噪声一起放大的概率；录音中仍可用 `W/S` 现场调大或调小。
- `processMicBuffer()` 新增 `recHum` 慢速低频底座跟踪器：每帧先做 DC/high-pass，再追踪机器自己的低频/偏置残留，并从样本里扣掉一部分。
- 静音或非人声段更积极扣低频底座；检测到类似人声时，追踪变慢、扣除比例降低，避免把字头和人声主体一起削掉。
- 没有使用 FFT、神经网络或重型降噪库；这是固件端“先少录进一点底噪”的保守方案，真正干净的人声仍主要依赖服务器端 v16 这种离线处理。

验证记录：直接用普通 `esp32:esp32:m5stack_cardputer` 分区编译会失败，因为程序 `1753348 bytes` 超过默认 `1310720 bytes` 上限；这不是本次改动导致的语法问题，而是该项目一直需要 8MB Flash + 3MB APP 分区。使用正式 FQBN `esp32:esp32:m5stack_cardputer:FlashSize=8M,PartitionScheme=default_8MB` 后编译通过：程序 `1753396 bytes`，占 `52%`；全局变量 `138200 bytes`，占 `42%`。

后续验证方式：目前只编译通过，尚未烧录。烧录后建议重新录一条“蒙在被子里”的静音样本和一条正常说话样本，对比服务器后台的原档/三档。若原始 WAV 本身底噪明显下降，说明源头治理有效；若声音偏小，优先用录音时 `W` 把 `recGain` 加回去，而不是先改服务器参数。

本轮主要继续修 CHAT 模式“像个正常助手”的体验，而不是继续堆规则。小猪字符画已经修到能保留换行、停留在屏幕上，并且 TTS 只读短确认，不朗读整张 ASCII 图。固件侧已经能显示用户输入 `YOU` 和助手回复 `REPLY`；回复播放结束后不再把正文变成 `READY`，`READY` 只画在右下角小状态位。

服务端已把 DeepSeek 链路按官方文档校正：线上模型使用 `deepseek-v4-pro`，CHAT 普通对话显式关闭 thinking，避免温度参数失效和等待变长；不再强制 JSON Output，减少空回复/半截 JSON 风险。规则层已经从大而宽的 `tryFastChatReply()` 收窄为 `tryEssentialChatReply()`：只保留绘图、简单算术、日期星期、明确实时信息不可用提醒，其余普通问题交给大模型自由回答。创意请求使用更高温度，事实/普通请求保持中低温。

针对“再来一个/换一个”误触发小猪规则的问题，服务端已加门槛：普通笑话、问题、创意追问不再被画图规则抢走；只有明确画图或非常短的绘图上下文 follow-up 才走 ASCII 规则。ASR 侧也做了收尾保护：实时识别必须明确句尾且不像半句话才采用，否则回退完整 WAV ASR，避免“小猪”尾字丢失后乱澄清。

TTS 体验已改为流式推送：服务器发出 `reply_text` 后立即开始请求 DashScope TTS PCM，并在首个音频块到达时发送 `audio_start` 和二进制音频帧，避免等完整 WAV 生成、落盘、读取、重采样之后才说话。若后续仍觉得语音慢，优先检查服务器 `tts_start/audio_start/audio_end` 时间差。

CHAT 右上角最终只保留电量，不显示时钟；电量颜色跟随 CHAT 主题色，低电量保持红色。CHAT 内 `Esc` 已改为清理上行、播放任务、喇叭、PCM ring、WebSocket 和 Wi-Fi 后返回 `APP_SLEEP` 息屏。

米家/Home Assistant 结论：Cardputer 本体不能装 Home Assistant，它是 ESP32 小设备，适合当遥控器。Home Assistant 要装在家里一个能常开的设备上，比如 Windows 电脑、旧笔记本、NAS、树莓派、x86 小主机、虚拟机或 Docker。未来如果要让小机器控制米家，建议架构是 `Cardputer -> 云服务器 -> 家里 Home Assistant -> 米家设备`。不建议让云服务器直接逆向登录米家云账号，风险和不稳定性都更高。

新窗口建议先验真机体验：`给我画一只小猪`、`换一个小猪`、连续让它讲笑话并说 `再来一个`、问生活类问题、确认用户输入和回复文字都可见、确认语音在文字出现后尽快开始、确认播放结束右下角显示 `READY`、确认 CHAT 内按 `Esc` 息屏。
