Cardputer CHAT 语音对话工作流备忘

本文记录“小机器用 C 键唤醒进入 CHAT，与助手语音对话”的工作流、职责边界和后续实施顺序。 第一版目标是稳定跑通按键语音对话，不做持续监听和实时双工。

当前决策

  • Space 继续作为息屏唤醒后快速录音入口。
  • F 继续作为息屏唤醒后进入番茄钟入口。
  • C 新增为息屏唤醒后进入 CHAT 语音对话入口。
  • CHAT 第一版采用“按键录一句、上传、等待回复、显示文字”的半双工流程。
  • 语音转文字、对话模型、命令解析和后续 TTS 都放在服务器做，不放在 ESP32-S3 本地做。

第一版必须做到

  • C 键唤醒直达 CHAT。
  • CHAT 内录一句话并上传。
  • 服务器返回助手文字回复。
  • 小机器屏幕显示回复摘要。

第一版暂不做

  • 持续监听唤醒词。
  • 实时边说边回。
  • 复杂多轮长上下文 UI。
  • 危险命令自动执行。

后续增强

  • 服务器生成 TTS，机器下载播放。
  • 后台网页可调助手提示词。
  • 安全命令白名单。
  • 短上下文记忆和最近录音检索。

唤醒入口

按键 唤醒后动作 说明
Space 进入录音 保留现有最快录音入口，不改变用户肌肉记忆。
Enter 进入录音列表 保留现有回放和管理入口。
F 进入番茄钟 与现有多应用唤醒设想一致。
C 进入 CHAT 新入口。唤醒后不自动录音，先显示 CHAT 准备界面，避免误触上传。

固件层可新增 APP_CHAT，并扩展 wakeAppFromPressedKeys()： 检测到 C/c 时返回 APP_CHAT。

CHAT 设备端流程

  1. 息屏时按 C，设备唤醒并进入 CHAT 页面。
  2. 页面显示简短状态：CHAT、网络状态、录音提示。
  3. 按 Enter 开始录音；再次按 Enter 结束并发送。
  4. 录音保存为临时或普通 WAV，可按后续策略决定是否留档。
  5. 固件通过 Wi-Fi 上传到服务器 CHAT 接口。
  6. 上传中显示 SEND / WAIT，允许 Esc 取消等待并返回息屏。
  7. 收到回复后，屏幕显示助手回复的短文本。
  8. 后续加 TTS 后，若服务器返回音频 URL 或音频文件，设备下载并播放。

服务器流程

服务器新增一条与录音备忘分开的 CHAT 路径，避免影响现有上传、转写、flomo 工作流。

Cardputer CHAT WAV/ADPCM
  -> POST /chat/upload
  -> 保存 chat 原始音频
  -> ASR 转文字
  -> 对话模型生成回复
  -> 可选：解析安全命令
  -> 可选：TTS 生成回复音频
  -> 返回 JSON 给设备

建议第一版返回结构：

{
  "ok": true,
  "id": "chat_20260526_153000",
  "userText": "帮我总结一下最近的录音",
  "replyText": "可以，我先看最近几条录音的转写结果。",
  "command": null,
  "audioUrl": null
}

固件和服务器分工

模块 负责内容 不负责内容
Cardputer 固件 按键唤醒、录音、上传、状态显示、播放服务器返回的音频。 大模型推理、高质量语音识别、TTS、复杂命令理解。
cloud-voice-server 接收音频、ASR、对话、TTS、命令白名单、上下文和日志。 实时采样、键盘事件、设备本地音频链路。
网页后台 测试 CHAT 请求、查看用户原文和助手回复、调提示词、回放 TTS。 替代设备端真实交互。

命令白名单

命令必须先从少量安全动作开始。模型只能提出结构化命令，固件或服务器只执行白名单内的动作。

命令 第一版建议 安全策略
show_recent_summary 可以做 只读最近录音转写，不改文件。
play_recording 可以做 只允许播放明确编号或最近一条。
mark_important 可以做 只标记当前或明确编号录音。
upload_recording 可以做 复用现有上传队列。
delete_recording 暂缓 必须二次确认，第一版不自动执行。
change_wifi_or_token 不做 配置类动作必须走设置页或电脑端配置工具。

删除、清空、覆盖配置、修改 token、批量移动文件等动作不得由模型一句话直接执行。

UI 草案

屏幕很小，CHAT 页面只放状态和短回复。长文本留在服务器后台。

┌────────────────────────┐
│ CHAT            WiFi OK │
│                        │
│ Press ENTER to talk    │
│                        │
│ Reply:                 │
│ 可以，我来帮你整理...   │
│                        │
│ ESC sleep        C new  │
└────────────────────────┘
  • Enter：开始/结束录音，或发送下一句话。
  • Esc：退出并息屏。
  • C：清空当前显示，准备新一轮对话。
  • Backspace：返回上一层应用列表，后续如果有应用列表再接入。

和现有录音系统的关系

  • CHAT 音频上传不要自动进入 flomo，避免把临时对话污染备忘录。
  • CHAT 原始音频是否保存到 SD，后续可做成配置。第一版建议保存最近一条临时文件，便于失败重试。
  • 服务器端 CHAT 日志单独放目录，例如 chat-jobs/、chat-audio/、chat-tts/。
  • 现有 /upload、DashScope 转写、DeepSeek 整理、flomo 推送路径不应被 CHAT 改动。
  • 如果复用现有 ASR 函数，应通过参数关闭 flomo 和备忘格式化。

实施顺序

  1. 先在文档中确认 C 键入口和 CHAT 半双工流程。
  2. 服务器新增 CHAT 接口和本地网页测试按钮，先用已有录音验证对话回复。
  3. 固件新增 APP_CHAT 和 C 键唤醒入口，先做占位页。
  4. 固件接入 CHAT 录音和上传，第一版只显示文字回复。
  5. 服务器加 TTS 音频生成，固件下载并播放回复。
  6. 加入少量只读或低风险命令白名单。
  7. 稳定后再考虑 Realtime 或更接近对讲机的低延迟模式。

待确认事项

  • 第一版 CHAT 音频是否保存在 SD：建议保留最近一条临时文件。
  • 第一版对话模型：可先复用服务器已有 DeepSeek；如果接 OpenAI，另行配置服务器环境变量。
  • 第一版是否立即加 TTS：建议先文字回复跑通，再加语音播放。
  • CHAT 是否带上下文：建议只保留最近 3 到 5 轮，防止请求过长。
  • 失败时是否落回普通录音：建议失败只显示错误，不自动转为 flomo 备忘。

电脑 Codex 桥接规划

当前已跑通的是“小机器 → 云端服务器 → 小机器显示 ASR 文本”。这还不是“小机器直接把消息发进当前 Codex 对话窗口”。 Codex 桌面应用目前没有给外部设备直接注入当前会话消息的稳定入口，因此需要做一层电脑端桥接。

  • 服务器侧新增 GET /api/chat/recent，电脑可用 X-Upload-Token 拉取最近 CHAT 语音文本。
  • 第一阶段桥接：电脑本地脚本轮询 /api/chat/recent，把最新 userText 写入本地 inbox 文件；需要时让 Codex 读取这个文件。
  • 第二阶段桥接：做一个本地小网页或托盘程序，显示小机器语音输入，并提供“一键复制/发送给 Codex”的工作流。
  • 真正自动进入当前 Codex 对话，需要 Codex 桌面端提供本地 API、插件入口或可控 MCP 通道；没有这个入口时，不建议用脆弱的 UI 自动化硬塞消息。
