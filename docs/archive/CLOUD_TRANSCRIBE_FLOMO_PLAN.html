Cardputer 录音自动上传、转写与 flomo 同步方案

目标：Cardputer 录完音后自动上传到阿里云服务器，服务器转成文字，再自动写入 flomo。本文档用于后续开发对齐。

  • 完整链路
    • 窗口 A：Cardputer 设备端
      • 负责人：设备端 AI 窗口
      • 盒子层负责 Wi-Fi 登录、NTP、HTTP 上传，不在小键盘上输入长 token
      • 录音保存为 WAV
      • 录音中按 \：结束录音并把当前 WAV 加入上传队列
      • 列表选中录音按 \：只上传选中的 WAV
      • 写入 /UPLOAD/queue.txt 待上传队列
      • 读取 /UPLOAD/net.txt 连接 Wi-Fi 和服务器
      • 联网成功后由盒子层 NTP 校时，给上传附带真实时间
      • 空闲时 HTTP POST 上传 WAV
      • 成功后写入 /UPLOAD/done.txt
    • 电脑配置页
      • 在电脑填写服务器 URL、上传 token
      • 生成 SD:/UPLOAD/net.txt
      • 阿里云 Key 和 flomo Webhook 只放服务器，不给小机器
    • 共同接口合同
      • 两边都必须遵守同一套 /upload 协议
      • Header: X-Upload-Token / X-Device-Id / X-Recording-Name / X-Recorded-At
      • Body: 原始 WAV bytes
      • 成功返回: {"ok":true,"id":"REC_XXXX"}
    • 窗口 B：阿里云服务器端
      • 负责端：服务器端 AI 窗口
      • /upload 接收 WAV 并校验 token
      • 保存 uploads/REC_XXXX.wav
      • 创建 jobs/REC_XXXX.json 记录状态
    • 窗口 B：阿里云转写
      • 调用 DashScope / Paraformer
      • 保存 transcripts/REC_XXXX.txt
    • 窗口 B：flomo
      • 服务器调用 Incoming Webhook
      • 生成 #Cardputer语音 memo
目标 当前状态 总链路 步骤链路 准备清单 服务器 设备端 电脑配置 实施阶段 安全 待确认

一、目标与边界

项目决定
设备端职责只负责录音、本地保存、排队上传 WAV，以及盒子层 Wi-Fi 登录 / NTP / HTTP。不要在 Cardputer 上转文字，也不要在小键盘上维护长 token、flomo webhook 或阿里云密钥。
服务器职责接收录音、保存文件、调用阿里云语音识别、生成文本、发送到 flomo。
电脑配置职责电脑网页负责填写服务器地址、上传 token、设备名，并生成 SD:/UPLOAD/net.txt。之后换 Wi-Fi 时，可以直接在小机器端改 ssid/password。
flomo 职责只接收服务器发来的文字 memo。flomo 自带语音输入不作为后台音频转写 API 使用。
性能原则上传必须后台化、可重试，不能影响录音、播放和列表操作。

层级状态说明
分区已完成已切到 FlashSize=8M,PartitionScheme=default_8MB，APP 上限从约 1.31MB 提高到 3.34MB。
编译尺寸已验证开启 Wi-Fi 上传骨架后：程序 1,729,020 bytes，占 51%；RAM 126,016 bytes，占 38%。
设备端架构已完成第一版用户按 \ 后，录音应用把当前/选中录音写入 /UPLOAD/queue.txt；盒子层提供 systemIdleTick() 作为空闲上传调度入口。Wi-Fi 可在小机器端修改，服务器长配置由电脑生成。
设备端 Wi-Fi骨架已进固件固件已包含轻量 WiFiClient 上传路径，只支持极简 HTTP POST；不在设备端做 HTTPS、阿里云 SDK、JSON 大解析。
设备端时钟骨架已进固件Cardputer 不依赖内置 RTC；盒子层在 Wi-Fi 连接成功后用 NTP 校时，上传时附带 X-Recorded-At。离线时仍可录音，只是没有真实时间戳。
服务器端未开始下一步要先写最小 /upload 服务，用电脑 curl 验证 WAV 能保存。
阿里云转写未开始服务器收到 WAV 后再接 DashScope / Paraformer，不让 Cardputer 直接调用云 API。
flomo未开始服务器转写完成后调用 flomo Incoming Webhook。

三、总链路

Cardputer 录音保存 WAV
  -> 用户按 \ 明确选择上传当前录音
  -> 写入 SD:/UPLOAD/queue.txt
  -> 盒子层在空闲时连接 Wi-Fi 并用 NTP 校时
  -> HTTP POST 上传到阿里云服务器 /upload
  -> 服务器保存 uploads/REC_XXXX.wav
  -> 服务器创建 jobs/REC_XXXX.json 状态
  -> 服务器调用阿里云 Paraformer / DashScope 转文字
  -> 服务器保存 transcripts/REC_XXXX.txt
  -> 服务器调用 flomo Incoming Webhook 写入 flomo
  -> 服务器把 job 标记为 done
  -> Cardputer 收到上传成功后写入 SD:/UPLOAD/done.txt
核心原则：Cardputer 只负责录音、排队、联网、轻量上传；电脑负责生成本地配置文件；服务器负责鉴权、HTTPS、转写、flomo、日志和复杂重试。

四、从现在开始的步骤链路

顺序动作验收标准
0大分区固件基线已烧录 8MB Flash + 3MB APP 分区版本；后续所有编译/烧录都使用新 FQBN。
1服务器最小上传接口阿里云服务器跑起 Node.js 服务；电脑用 curl 上传 WAV，服务器保存到 uploads/。
2服务器鉴权与状态文件/upload 校验 X-Upload-Token；保存 jobs/REC_XXXX.json，能通过 /jobs/:id 查看。
3电脑生成服务器配置在电脑打开 toolbox/NET_CONFIG_HELPER.html，填写服务器 URL、token、device id，生成 /UPLOAD/net.txt 放到 SD 卡；首次 Wi-Fi 也可在电脑填，之后可在小机器 Wi-Fi 设置里改。
4Cardputer 上传联调录音中按 \ 或列表选中按 \ 后入队；空闲时连 Wi-Fi 上传；服务器收到真实录音 WAV；成功后本地写 /UPLOAD/done.txt。
5失败重试与 UI 状态断网时不影响录音；重新进入列表/工具箱后继续上传；列表能看出待传/已传/失败。
6接阿里云转写服务器把 WAV 发给 DashScope / Paraformer，拿到文字并保存 transcripts/REC_XXXX.txt。
7接 flomo转写成功后自动写入 flomo，memo 格式稳定且含录音编号/时间/来源。
8端到端稳定性连续录多条、断网、重启、删除普通录音、快捷/重要录音都不破坏上传链路。

六、服务器设计

目录结构建议

/opt/cardputer-voice/
├── server.js
├── package.json
├── uploads/       # 原始 WAV
├── transcripts/   # 转写文本
├── jobs/          # 状态 JSON
└── logs/

环境变量

PORT=3000
UPLOAD_TOKEN=一串随机长密码
DASHSCOPE_API_KEY=阿里云语音识别key
FLOMO_WEBHOOK_URL=https://flomoapp.com/iwh/...
PUBLIC_BASE_URL=https://你的域名

接口规划

接口方法作用
/healthGET服务器健康检查，Cardputer 可用它测试网络。
/uploadPOST上传 WAV，Header 带 X-Upload-Token。
/jobs/:idGET查看某条录音的处理状态，调试用。

上传请求格式

POST /upload HTTP/1.1
Headers:
  X-Upload-Token: UPLOAD_TOKEN
  X-Device-Id: cardputer-001
  X-Recording-Name: REC_0123.wav
  X-Recorded-At: 2026-05-24 14:30:00
Content-Type: audio/wav

Body:
  WAV bytes

第一阶段 curl 验证

curl -X POST "http://你的服务器:3000/upload" \
  -H "X-Upload-Token: UPLOAD_TOKEN" \
  -H "X-Device-Id: cardputer-001" \
  -H "X-Recording-Name: REC_0001.wav" \
  -H "Content-Type: audio/wav" \
  --data-binary "@REC_0001.wav"

七、Cardputer 端设计

模块做法
分区/FQBN必须使用 esp32:esp32:m5stack_cardputer:FlashSize=8M,PartitionScheme=default_8MB。
上传时机不自动上传所有录音。录音中按 \ 会结束录音并入队；列表中选中录音按 \ 只上传该录音；盒子层空闲时上传。
失败处理上传失败保留本地 WAV，下次开机/进入列表/空闲时重试。
Wi-Fi 设置工具箱里新增 Wi-Fi 应用：可在小机器上输入/修改 Wi-Fi 名和密码，保存后测试连接。它会保留电脑配置页写入的服务器 URL 和 token。
配置文件/UPLOAD/net.txt 只放在 SD 卡，不进 GitHub，不写死进源码。服务器 URL/token 由电脑配置页生成；小机器只维护 Wi-Fi。
状态记录/UPLOAD/queue.txt 记录待上传；/UPLOAD/done.txt 记录已上传。
盒子时钟联网成功后用 NTP 校时；如果录音时已经有时间，就记录到 /UPLOAD/recorded_at.txt；上传时通过 X-Recorded-At 发给服务器。
目录关系普通、快捷、重要录音都可以上传；上传不应改变三列目录分类。
性能约束播放中不上传；录音中不上传；只在空闲、列表或保存后短窗口尝试。

SD 卡配置文件格式

# SD:/UPLOAD/net.txt
ssid=你的WiFi名
password=你的WiFi密码
url=http://你的服务器:3000/upload
token=一串随机长密码
device=cardputer-001
ntp=pool.ntp.org
tz=8
当前设备端轻量上传只按 http:// 设计。公网正式入口可以由服务器/Nginx 负责 HTTPS；Cardputer 到服务器先走内网或测试 HTTP，避免把 TLS 复杂度塞进固件。

八、电脑配置页

文件作用
toolbox/NET_CONFIG_HELPER.html在电脑浏览器填写服务器 URL、上传 token、设备名、NTP 和时区，自动生成 net.txt 内容。Wi-Fi 可先在电脑填，也可之后在小机器端改。
SD:/UPLOAD/net.txt小机器读取这个文件后才能联网和上传。小机器不保存阿里云 Key、不保存 flomo Webhook。
当前最省力的操作方式：服务器窗口给出 url 和 token；本窗口在电脑配置页生成 net.txt；把它放到 SD 卡 /UPLOAD/。以后到了新地方，只进小机器工具箱的 Wi-Fi 修改网络即可。

十、安全原则

不要把 DASHSCOPE_API_KEY、FLOMO_WEBHOOK_URL、UPLOAD_TOKEN、Wi-Fi 密码提交到 GitHub。
  • flomo webhook 只放服务器环境变量。
  • 阿里云 API Key 只放服务器环境变量。
  • Cardputer 固件不写死任何密钥；小机器侧只读取电脑生成的 /UPLOAD/net.txt。
  • 小机器不保存阿里云 API Key 和 flomo Webhook，这两项只在服务器环境变量里。
  • 服务器接口必须校验 X-Upload-Token。
  • 服务器日志不要打印完整 webhook 和 API Key。
