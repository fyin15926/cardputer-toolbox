Cardputer Toolbox 多应用架构建议

目标是把当前录音应用扩展成一个按键驱动的小工具系统。录音、番茄钟、设置、快捷音频等应用应共享同一套启动器和系统层，但运行时尽量互相隔离。

结论

推荐继续做大，但不要在现有录音流程里继续堆功能。应先抽出一个很薄的“系统壳”：负责息屏唤醒、按键路由、应用列表、全局音量、电量显示和资源占用规则。

单个应用不会因为应用数量变多而明显变慢。真正影响速度的是后台任务、频繁扫描 SD、资源没有释放、UI 每帧重绘过多，以及 Mic/Speaker/SD 被多个功能同时争用。

保留快速入口

  • Space：快速录音
  • Enter：录音列表
  • F：番茄钟
  • Ctrl：应用列表

一次只运行一个应用

录音、播放、番茄钟、设置页都作为前台应用运行。除非非常必要，不做复杂后台常驻任务。

录音资源独占

录音和播放期间，其他应用不要动 Mic、Speaker、SD 的状态。

推荐启动逻辑

按键 唤醒后动作 备注
Space 进入录音 保留现在最快的录音入口。
Enter 进入录音列表 适合回放、管理、绑定快捷键。
F 进入番茄钟 建议先做占位界面，再补完整计时逻辑。
Ctrl 进入应用列表 显示应用名、快捷键、状态，支持上下选择和回车进入。
其他字母 预留给未来应用 例如便签、计时器、文件管理、设置等。

代码结构建议

先不必大规模拆文件，但应在代码层面建立应用边界。

enum AppId {
  APP_NONE,
  APP_REC_RECORD,
  APP_REC_LIST,
  APP_POMODORO,
  APP_LAUNCHER,
  APP_SETTINGS
};

struct AppEntry {
  char key;
  const char *name;
  AppId app;
};

static const AppEntry APPS[] = {
  {' ', "REC", APP_REC_RECORD},
  {'\n', "LIST", APP_REC_LIST},
  {'f', "POMO", APP_POMODORO},
  {0, "APPS", APP_LAUNCHER}
};

主循环只做三件事：等待唤醒、解析按键、启动对应应用。应用内部结束后返回系统层，由系统层决定回息屏还是回应用列表。

资源规则

  • 进入录音前调用 prepareMicInput()，并清掉扬声器状态。
  • 进入播放前调用 speakerOn()，并清掉麦克风状态。
  • 普通应用不要直接改 ES8311 寄存器，除非它是音频应用。
  • SD 目录扫描只在进入列表、录音完成、删除、绑定后进行，不要每帧扫描。
  • 应用切换时清理长按删除、按键继承、临时 UI 提示等状态。

番茄钟可以常驻计时状态，但 UI 不应常驻刷新。息屏时只保留目标结束时间或剩余秒数，唤醒后再重绘。

番茄钟建议

  • 第一版只做：25 分钟工作、5 分钟休息、开始/暂停/重置。
  • F 唤醒直达番茄钟。
  • Enter 开始/暂停，Del 长按重置，Esc 返回息屏。
  • 计时建议用 millis() 差值，不依赖真实时钟。
  • 如果未来需要跨断电保存，再把番茄钟状态写入 SD。

录音应用后续整理

录音应用已经比较复杂，建议后续逐步整理，而不是一次重写。

  1. 先保留现有录音、播放、列表逻辑。
  2. 把入口从 loop() 迁移到统一启动器。
  3. 把录音相关全局状态加上 rec 前缀或集中到结构体。
  4. 把文件扫描、快捷键、分类 KEY/IMP 做成录音模块内部服务。
  5. 最后再考虑拆成多个 .h/.cpp 文件。

性能判断

  • 不会明显变慢：多个应用只存在于固件里，一次只运行一个。
  • RAM 目前还够：当前 RAM 使用约 30%，空间比较宽裕。
  • Flash 要留意：当前程序区约 88%，新增大功能时要控制代码体积。
  • 最该避免：后台同时跑多个任务、频繁扫 SD、多个应用争用音频资源。

建议实施顺序

  1. 新增应用枚举和应用表。
  2. 新增应用列表界面，通过 Ctrl 进入。
  3. 保留 Space 和 Enter 当前行为。
  4. 新增 F 进入番茄钟占位页。
  5. 番茄钟占位页稳定后，再补完整计时功能。
  6. 最后再整理录音模块内部结构。
